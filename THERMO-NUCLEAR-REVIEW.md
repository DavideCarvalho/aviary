# Thermo-Nuclear Review: aviary

Aviary is **not** a library — it's the documentation site for the `@dudousxd/nestjs-*` ("Aviary") ecosystem: a Fumadocs/Next.js 16 app configured for static export to GitHub Pages, with Orama static search, per-library doc folders synced from sibling repos, and LLM-text routes (`llms.txt`/`llms-full.txt`/`llms.mdx`). The hand-written TypeScript surface is tiny (~250 LOC across `lib/` + a handful of `app/` route handlers); the bulk of the LOC count is generated (`.source/`, `.next/types/`) and should be excluded from any code-quality judgement. There are no significant code-quality issues in the authored code.

## Top findings

### 1. Most of the counted LOC is generated, not authored — INFORMATIONAL
`.source/server.ts` (213), `.next/types/validator.ts` (142), `.source/browser.ts`, `.source/dynamic.ts`
These are Fumadocs/Next codegen outputs (`fumadocs-mdx`, `next typegen`), regenerated by `postinstall`/`predev`/`prebuild`. They are correctly gitignore-able build artifacts and must not be reviewed as hand-written code. The actual authored TS is `lib/*.ts` (~225 LOC) plus thin `app/**/route.ts` handlers. Verdict on the authored surface: clean and idiomatic.

### 2. `lib/libs.ts` carries a duplicated icon as both component and string — LOW (boundary)
`aviary/lib/libs.ts:30-34` (the `icon: LucideIcon` + `iconName: string` pair on every entry)
Each library entry stores both `icon: ShieldCheck` (the component, for cards) and `iconName: 'ShieldCheck'` (the string, for `meta.json` sidebar tabs). The two must be kept in sync by hand — a classic dual-source-of-truth. The `lib/source.ts` `icon()` resolver already maps a string name back to a lucide component against the full namespace.
Remedy: keep only `iconName: string` and derive the component where needed via the same `lucideExports[iconName]` lookup `source.ts` uses (or a tiny `iconFor(name)` helper). Removes the per-entry sync hazard. Low priority — it's a static, rarely-edited catalogue.

### 3. `app/llms-full.txt/route.ts` maps then awaits — fine, but worth noting — LOW (orchestration)
`aviary/app/llms-full.txt/route.ts:7-8`
`source.getPages().map(getLLMText)` then `await Promise.all(scan)` is correct concurrent fan-out (not sequential). No change needed; flagged only to confirm it's already the right pattern. `getLLMText` does one `await page.data.getText('processed')` per page with no shared mutable state.

## Largest files
No authored file exceeds 600 LOC. `lib/libs.ts` is the largest hand-written file at 155 LOC and is a flat data catalogue (appropriate). The only >600-style concern would be generated files, which are out of scope.

## What's good
- Correct framework idioms throughout: `revalidate = false` + `generateStaticParams` on the dynamic markdown route is exactly right for static export; the search route uses Fumadocs' `staticGET`.
- The `basePath`/GitHub-Pages story is handled centrally (`PAGES_BASE_PATH`, `lib/shared.ts` route constants) rather than sprinkled — and the README documents the gotcha and the cron/manual/`repository_dispatch` doc-sync triggers clearly.
- `lib/source.ts`'s lucide resolver deliberately resolves against the full namespace (not the canonical `icons` map) so renamed aliases like `Filter`→`Funnel` still work, with a comment explaining why. Thoughtful.
- `lib/cn.ts` is a one-line re-export of `twMerge` — a legitimate thin alias, not over-abstraction.
- Clean separation of concerns: data (`libs.ts`), source/loader config (`source.ts`), shared constants (`shared.ts`), and route handlers each do one thing.
